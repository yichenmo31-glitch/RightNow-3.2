# /fankui 技能规则（必须严格遵守）

## 触发条件
当用户输入 `/fankui` 或 `/fankui [任意指令]` 时，立即进入「问题反馈优化模式」。

## 第一步：强制扫描文件夹
- 项目根目录下必须存在 `./问题反馈/` 文件夹（如果不存在，立即提醒用户创建）。
- 使用工具列出 `./问题反馈/` 及其所有子文件夹的**全部文件**（支持嵌套）。
- 优先处理**最新修改**的文件（按文件修改时间排序）。

## 第二步：智能分析所有内容
对每个文件进行以下处理：

**文本文件（.txt, .log, .md, .py, .js, .ts, .json, .yaml 等）**
- 完整读取内容
- 提取关键信息：报错栈、日志、用户描述、代码片段
- 总结核心问题（用 bullet points）

**图片文件（.png, .jpg, .jpeg, .gif, .webp, .heic）**
- 使用 Vision 能力**详细描述**图片内容
- 重点关注：
  - UI 界面问题（布局错位、样式异常）
  - 错误弹窗/Toast
  - 崩溃截图、控制台报错
  - 代码截图中的具体行号
- 描述要精确到能让别人看懂问题

**其他文件**
- 视频/音频暂不支持，提示用户转为截图

## 第三步：输出标准化格式
1. **📁 文件清单**（列出所有已分析文件）
2. **🔍 问题总结**（合并所有文件/图片的核心问题）
3. **💡 用户指令解析**（如果用户带了指令，先复述）
4. **🚀 优化方案**（分步、可执行）
   - 给出具体代码修改建议（用 ```diff 格式）
   - 优先提供**最小改动**方案
   - 附上完整可复制的新文件内容（必要时）
5. **❓ 确认下一步**（主动询问用户要先改哪一块）

## 后续对话规则
- 此后所有对话都保持在「问题反馈优化模式」中
- 用户说"继续""下一个""改这里"时，直接执行对应优化
- 每次修改后自动生成 diff 并说明改动理由
- 优化方向优先级：修复 bug > 性能 > 可读性 > UX > 新功能

## 禁止事项
- 不要问"文件夹在哪里"——直接用工具读
- 不要输出"已读取"这种废话，直接给分析结果
- 图片描述必须客观，不能猜测用户意图

## 注意事项
- 所有沟通使用中文
- 遵循项目代码规范（见 CLAUDE.md）
- 样式使用 Tailwind CSS，深色主题（#030303 背景，#B8FF00 主色）
- 移动端优先
- 修改前先读取目标文件，理解现有代码再动手
- 不要过度工程化，只修复反馈中提到的问题
