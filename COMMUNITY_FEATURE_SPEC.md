# RightNow 社区模块功能文档

## 1. 背景

RightNow 的主产品链路，已经在“看见理想自己”和“开始被 AI 教练带着走”这两部分建立了基础。

但如果产品要真正帮助用户长期坚持，仅靠个人目标和 AI 教练还不够，还需要一个外部动力系统。

社区模块的作用，就是补上这个“外部动力”：

- 让用户的进步被看见
- 让用户获得同频陪伴
- 让用户在关系和反馈中更容易坚持

因此，RightNow 的社区，不应该是一个普通的发帖区，也不应该是泛社交广场。

它的本质，是一个服务“长期健身坚持”的内容 + 关系 + AI 协作系统。

## 2. 产品目标

社区模块的核心目标，是通过“晒成果 + 找搭子 + 激励坚持”的组合机制，增强用户持续行动的外部驱动力。

优先级明确如下：

1. 晒成果（第一优先级）
2. 找搭子（第二优先级）
3. 激励坚持（第三优先级）

这个模块成功的标志是：

- 用户愿意把自己的变化和阶段成果展示出来
- 社区首页能持续看到“可被感知的进步”
- 用户能够通过 Agent 找到更适合一起坚持的搭子
- 搭子关系不只是被建立，而是被持续维护
- AI 在社区里能同时承担内容整理、关系撮合、节奏维护的角色

## 3. 核心产品定位

RightNow 社区的第一性目标，不是聊天，而是让“进步”变成值得展示、值得互动的内容。

它应该：

- 让用户快速晒出自己的成果
- 让别人看见变化并获得激励
- 让系统帮助用户找到一起坚持的人
- 让 AI 把内容和关系都运营起来

它不应该：

- 变成一个泛话题社区
- 被日常碎片动态淹没
- 把重心放在无目的聊天上

## 4. 首页主结构

社区首页建议采用双层主结构：

### 4.1 顶部：搭子推荐区

首页顶部展示轻量的搭子推荐入口。

目标是：

- 提前露出“适合你一起坚持的人”
- 让找搭子成为首页即可感知的第二主能力
- 但不抢走“晒成果主 feed”的主视觉位置

用户点击后，进入完整的搭子页。

### 4.2 中部主区：进步动态流

社区首页的主视觉和主停留区，应当是“进步动态流”。

这个主 feed 的目标不是展示碎片流水，而是展示：

- 前后对比
- 阶段成果
- AI 整理后的进步总结

也就是说，用户进入社区第一眼看到的，应该是别人“真的在变好”的证据。

## 5. 核心内容：进步报告卡

社区的核心内容单元，不应是普通帖子卡，而应是“进步报告卡”。

它是整个社区最重要的内容形态，用来把用户的变化包装成值得被看见的内容。

### 5.1 卡片必备信息层

一张进步报告卡，建议固定包含以下四层信息：

1. 主视觉
   - 前后对比图
   - 或阶段成果图

2. AI 总结标题
   - 一句概括变化的标题

3. 关键变化指标
   - 体重变化
   - 训练完成次数
   - 连续打卡天数
   - 其他可量化阶段成果

4. 用户感受
   - 一句自己的体感、情绪或阶段反馈

### 5.2 内容节奏

社区内容节奏采用混合型：

- 日常可以发
- 但系统应更鼓励阶段性高质量成果帖

这样既保留活跃度，也避免社区主 feed 被低价值流水内容稀释。

## 6. AI 第一优先能力：帮用户发得更好

AI 在社区里的第一优先能力，应是“帮用户把成果发得更好看”。

这也是社区里最容易被用户直观感知到的价值。

### 6.1 标准发帖链路

1. 用户完成打卡、训练，或出现阶段性变化
2. 日常场景下，用户可主动点击“帮我发”
3. 阶段性高价值节点，系统可主动提醒“这次变化值得发一条”
4. AI 生成一条“进步报告型”草稿
5. 用户进入确认页
6. 用户可编辑并确认
7. 用户确认后再发布
8. 内容进入社区主 feed

### 6.2 用户可编辑项

发帖前，用户必须可以编辑：

- 文案
- 图片（可换图 / 不发图）
- 公开范围

### 6.3 公开范围建议

第一版建议支持以下三档：

- 全部可见
- 仅相近目标人群可见
- 仅搭子可见

这个能力很重要，因为它直接影响用户是否愿意发帖。

### 6.4 信任边界

必须明确：

- AI 不直接替用户发帖
- 所有内容都先生成草稿
- 用户确认后才发布

这样既保留 AI 的高价值，又不会让用户产生“被越权代发”的不适感。

## 7. 找搭子机制

社区的第二优先能力，是“找搭子”。

但它不应是普通好友推荐，而应该是一个围绕坚持节奏的 AI 陪练匹配系统。

### 7.1 匹配维度

匹配采用三者结合，但优先级如下：

1. 节奏匹配（第一优先）
   - 每周训练频率
   - 常见训练时段
   - 活跃节奏

2. 目标匹配
   - 减脂
   - 增肌
   - 重塑

3. 情绪 / 风格匹配
   - 喜欢被严格督促
   - 喜欢轻松陪伴
   - 喜欢游戏化激励

### 7.2 搭子入口

搭子入口采用双入口设计：

- 社区首页顶部轻量推荐区
- 独立完整搭子页

### 7.3 搭子推荐卡建议字段

第一版推荐卡建议至少展示：

- 目标方向
- 训练节奏
- 最近活跃度
- AI 推荐理由

## 8. AI 在搭子中的角色

Agent 在搭子系统中，不应只做一次性推荐，而应承担轻运营角色。

它至少负责三件事：

1. 推荐适合的搭子
2. 帮双方完成破冰
3. 后续提醒两个人一起打卡、一起坚持

这意味着，AI 不只是撮合关系，而是在帮助维护关系。

## 9. 关系容器：双人小房间

每一段搭子关系，不应只是一个“好友状态”，而应该进入一个独立的双人小房间。

因为用户未来可能同时拥有多个搭子，所以每一段搭子关系都需要自己的关系空间。

双人小房间的本质是：

- 一个以共同执行为核心的关系空间
- 一个由 AI 持续维护节奏的轻量训练房间

它不是普通聊天房，也不是纯任务页。

## 10. 双人小房间第一版结构

### 10.1 顶部：双方状态面板

这是小房间首页的第一优先内容。

应一眼展示：

- 我今天完成了没
- 对方今天完成了没
- 各自最近连续完成天数
- 当前谁在掉队，谁在维持节奏

这个区域的价值是：

- 制造轻压力
- 强化陪伴感
- 不需要阅读长文案就知道当前状态

### 10.2 中部：AI 教练共同引导区

AI 作为共享教练，在这里发起：

- 今日共同任务
- 阶段鼓励
- 掉队拉回
- 明日目标预告

因为“双方状态面板”是第一优先，所以 AI 区应服务状态，而不是抢占主视觉。

### 10.3 底部：轻互动区

第一版不做重聊天，只做轻互动。

建议包括：

- 点赞 / 加油
- 戳一戳
- 预设短句
  - “我练完了”
  - “你也快去”
  - “今天一起完成”
  - “明天继续”

这样既保留关系感，又不会把产品复杂度和运营成本拉得太高。

## 11. 掉队时的 AI 处理方式

如果双人关系里，其中一人掉队：

- AI 应同时提醒两个人
- 但两侧文案不同

对掉队的人：

- 轻拉回
- 给台阶
- 必要时降低任务门槛

对已经完成的人：

- 先给予认可
- 再提示对方还没跟上
- 鼓励用一个轻互动把对方拉回来

这个机制的目标是：

- 让 AI 更像共享教练
- 让关系本身成为坚持的推动力
- 避免系统只做单点催促

## 12. 多 Agent 执行方案

这个模块同样应按多 Agent 协作来推进。

### 12.1 Antigravity

Antigravity 负责社区模块的 UI/UX 和前端体验主导。

主要范围：

- 社区首页整体结构
- 进步报告卡
- AI 发帖确认流
- 搭子推荐区
- 搭子页
- 双人小房间
- 轻互动区

Antigravity 是以下内容的 owner：

- 视觉层级
- 信息卡片形态
- 动效和交互节奏
- 社区模块的生产级前端体验

### 12.2 Claude Code / Codex

Claude Code / Codex 负责后端逻辑、状态流和实现审查。

主要范围：

- 帖子结构与公开范围规则
- AI 发帖草稿与确认流的数据逻辑
- 搭子匹配规则
- 双人关系状态结构
- 小房间状态同步
- AI 共同打卡提醒逻辑
- 边界情况和集成正确性审查

Claude Code / Codex 是以下内容的 owner：

- 数据结构
- 逻辑正确性
- 状态流设计
- 实现审查与可靠性

### 12.3 产品协同顺序

建议的推进顺序：

1. 先由 Antigravity 产出社区首页、进步报告卡、搭子区和双人小房间的前端方案。
2. 再由 Claude Code / Codex 按确认后的交互，定义最小后端结构和状态逻辑。
3. 最后从产品视角复核，确认这套系统是否真的服务于“晒成果、找搭子、坚持下去”。

## 13. PRD 用户故事

### 13.1 晒成果

作为一个用户，我希望能把自己的阶段成果整理成一条更有说服力的内容发出去，这样我会更愿意展示自己的进步。

### 13.2 被激励

作为一个用户，我希望在社区里看到别人真实的变化和阶段成果，这样我会更有动力继续坚持。

### 13.3 找搭子

作为一个用户，我希望系统能帮我找到节奏更合适的搭子，而不是让我自己在社区里盲找，这样我更容易建立稳定关系。

### 13.4 被持续拉着走

作为一个用户，我希望搭子关系不是加完就结束，而是有 AI 帮我们持续维护节奏，这样关系才会真的有价值。

## 14. 验收标准

### 14.1 社区首页

- 社区首页主区为进步动态流
- 首页顶部可见搭子推荐区
- 首页主视觉重点仍在“可被感知的进步”

### 14.2 进步报告卡

- 主卡片必须展示前后对比 / 成果图、AI 标题、关键指标、用户感受
- 社区内容应支持日常发帖和阶段性高质量成果帖共存

### 14.3 AI 发帖

- 用户可以手动点击“帮我发”
- 阶段性高价值节点可被系统主动提醒
- AI 生成的是草稿，不直接发布
- 用户必须可编辑文案、图片、公开范围后再确认发布

### 14.4 搭子系统

- 首页顶部可见搭子推荐入口
- 搭子匹配以节奏匹配为主
- 推荐卡可展示目标方向、训练节奏、活跃度、AI 推荐理由

### 14.5 双人小房间

- 每段搭子关系都有独立双人小房间
- 小房间首页第一优先展示双方状态面板
- 小房间支持 AI 共同任务和轻互动
- 掉队时，AI 能同时提醒两个人，并使用不同文案

## 15. 交付阶段

### Phase A：前端优先探索

负责人：Antigravity

目标：

- 先把内容展示、搭子露出、小房间关系空间做成真实可感知的前端体验

交付物：

- 社区首页首版
- 进步报告卡
- AI 发帖确认流
- 搭子推荐区与搭子页
- 双人小房间首版

### Phase B：产品复核与迭代

负责人：产品方向主导，Antigravity 配合

目标：

- 基于真实前端方案，调整首页主次关系、卡片信息密度和关系感表达

交付物：

- 修订后的社区前端结构
- 固化后的关键交互决策

### Phase C：后端映射与集成

负责人：Claude Code / Codex

目标：

- 在前端方案稳定后，补上最小可用的数据和状态逻辑

交付物：

- 精简帖子结构
- 发帖草稿流与可见范围逻辑
- 搭子关系状态
- 双人小房间状态流

## 16. 实施说明

- 社区模块同样应遵循“前端先行”的推进方式。
- 先看得到社区的内容感、关系感和节奏感，再决定后端要做多重。
- 当前阶段做后端规划的目的，是避免未来返工，而不是提前锁死产品。

## 17. 页面 / 组件清单

为了方便前端拆活，第一期建议明确以下页面和组件：

1. 社区首页顶部搭子推荐区
2. 社区首页进步动态流
3. 进步报告卡
4. AI 发帖提醒入口
5. AI 发帖草稿确认页
6. 搭子推荐卡
7. 搭子页
8. 双人小房间顶部状态面板
9. 双人小房间 AI 引导区
10. 双人小房间轻互动区

这些组件应作为一个完整的社区动力系统来设计，而不是拆成孤立的页面功能。
