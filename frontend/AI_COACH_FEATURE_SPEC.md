# RightNow AI 教练功能文档

## 1. 背景

RightNow 目前已经完成了用户旅程的前半段：

- 产品已经采集了年龄、身高、体重等基础身体信息。
- 用户已经通过 AI 共创出了自己的理想身材结果。
- 系统已经可以基于用户现状与理想身材，大致推断当前体脂率和目标体脂率。

但当前还缺少用户旅程的后半段：

- 从“我看到了未来的自己”变成“我知道今天该做什么”
- 从一次性的 AI 生图时刻，变成持续的教练陪伴关系
- 从被动问答，变成主动、个性化、按天推进的行动指导

AI 教练模块的作用，就是补上这一段闭环。

它不应该像一个通用聊天机器人，而应该同时承担三种角色：

- 专业训练顾问
- 每日督促者
- 情绪陪伴型健身搭子

整个模块的体验应该让用户感觉：

- 这个教练已经理解我了
- 它不是从零开始重新盘问我
- 它是真的会带着我往下走

## 2. 产品目标

AI 教练模块的核心目标，是把用户的理想身材愿景，转化成可执行、可坚持的训练节奏。

这个模块成功的标志是：

- 用户在理想身材生成后，能立刻进入一段有个性化感的首次教练体验
- AI 通过可见的分析和现实的节奏预期建立信任
- AI 只采集最少但足够启动教练关系的信息
- AI 能立刻给出清晰的首日行动计划
- AI 能用轻量但持续的方式完成前 7 天的跟进

## 3. 核心产品定位

AI 教练是嵌入在首页里的主动型模块。

它应该：

- 通过现有悬浮 AI 机器人主动出现
- 优先使用系统已经掌握的用户数据，再补充追问
- 用专业但有温度的方式引导用户
- 对不合理选择做轻柔纠偏，而不是被动接受

它不应该：

- 像一个普通客服式聊天页
- 一上来就抛给用户一长串问卷
- 依赖用户自己去发现并点开功能

## 4. 首次进入流程

### 4.1 入口

首页现有的悬浮 AI 机器人，就是这个模块的主入口。

- 不新增首页主按钮
- 由机器人主动触发首次教练交互
- 复用现在已有的红点 / 未读提醒能力

### 4.2 首次主动对话

第一次交互应该拆成 2 到 3 条消息，而不是一次性发成长段内容。

推荐消息顺序：

1. 一条短开场，告诉用户 AI 已经基于他的现状和目标做了初步分析。
2. 一张结构化的“初始身体判断报告卡”。
3. 一段健康周期说明，并自然过渡到首次教练建联。

## 5. 初始身体判断报告

这张报告卡是整个功能的信任锚点。

它不是医学报告，而是一份 AI 给出的起始判断，要求同时具备：

- 个性化
- 透明感
- 可校准

### 5.1 必须展示的字段

- 当前体脂率预估
- 目标体脂率预估
- 目标方向（减脂 / 增肌 / 重塑）
- 基础代谢预估（BMR）
- BMI 预估
- 每日总热量消耗建议（TDEE）
- 一句 AI 阶段判断

### 5.2 周期判断逻辑

系统还应该根据用户现状与目标之间的差距，给出一个健康理论周期。

例如：

- 如果当前预估体脂是 20%，目标是 15%，AI 应该给出一个合理、健康的周期，而不是默认用户用过激节奏去理解目标

系统应该做到：

- 给出一个健康理论周期建议
- 允许用户沿着这个建议节奏继续
- 允许用户后续用健身房或 InBody 数据手动校准

注意：

- “推荐”应该通过 AI 文案表达
- 不要在按钮文案里直接写“推荐”

## 6. 最小补充信息采集

在用户继续后，AI 只采集启动教练关系所需的最小信息。

第一批问题以按钮为主，只在必要的地方开放文本输入。

### 6.1 第一批必采字段

1. 训练基础
   - 零基础
   - 入门（偶尔练）
   - 中级（持续训练）
   - 高级（有系统训练经验）

2. 伤病或运动限制
   - 没有明显伤病或限制
   - 有，需要告诉你

如果用户选择“有”，则展开：

- 膝盖
- 腰
- 肩颈
- 心血管
- 其他

并提供：

- 可选的补充说明输入框

3. 每周训练频率
   - 1-2 天
   - 3 天
   - 4 天
   - 5 天及以上

特殊规则：

- 如果用户选择 1-2 天，AI 不应直接接受
- AI 先轻微调侃，再进行专业纠偏
- 明确提示当前目标通常至少需要 3 天训练频率
- 然后要求用户重新选择

4. 单次训练可投入时长
   - 15-20 分钟
   - 30 分钟
   - 45 分钟
   - 60 分钟以上

## 7. 首日输出

完成第一批补采后，AI 应该立刻生成一个“轻量但完整”的首日教练结果。

### 7.1 必备结构

1. 今日训练任务
   - 使用标准训练卡形式，包含动作、组数、次数和时长

2. 今日饮水 / 饮食提醒
   - 给出轻量、可执行的提醒，不做复杂营养方案

3. 今日打卡动作
   - 输入一句感受
   - 可选上传照片

这个结果应该让用户感受到“今天就可以照着做”，而不是泛泛而谈的建议。

## 8. 首周 7 天教练节奏

前 7 天的体验目标，是建立行动惯性，但不能做得烦人。

### 8.1 消息节奏

- 第 1-3 天：更主动
- 第 4-7 天：根据用户行为智能触发

### 8.2 核心触发信号

- 今日训练 TODO 是否完成
- 用户是否完成打卡
- 用户是否长时间未响应
- 用户是否连续完成了 2-3 天任务

### 8.3 用户掉队时的处理方式

如果用户没练，也没回复：

- 先用搭子式轻调侃把人拉回来
- 再用专业语气做纠偏
- 最后给一个更容易完成的替代任务，而不是只施压

### 8.4 第 7 天总结

第 7 天的总结必须围绕“成就感”来设计。

成就感的来源应包括：

- 完成次数
- 用户自己的感受反馈
- AI 的认可和鼓励

这个总结应该：

- 让用户看到自己真的做成了什么
- 让进步变得可感知
- 在庆祝后，轻量过渡到下一周

它不应该像一张冷冰冰的数据导出页。

## 9. 交互体验原则

AI 教练的感受应该是：

- 专业
- 有温度
- 主动
- 略带轻松感
- 不幼稚
- 不像客服机器人

交互设计上优先：

- 默认使用短消息
- 在关键节点使用信息卡片
- 优先使用按钮流
- 尽量减少用户的表单疲劳

## 10. 多 Agent 执行方案

这个功能必须按多 Agent 协作方式来执行，而不是混在一个模糊流程里做。

### 10.1 Antigravity

Antigravity 负责 UI/UX 和前端交互主导。

主要范围：

- 首页悬浮 AI 教练入口体验
- 首次主动消息节奏与动效
- 初始身体判断报告卡 UI
- 按钮式问题卡
- 纠偏状态下的对话 UI
- 首日 TODO 卡 UI
- 第 7 天成就感总结卡

Antigravity 是以下内容的 owner：

- 视觉方向
- 动效与交互细节
- 该模块的生产级前端体验

### 10.2 Claude Code / Codex

Claude Code 和 Codex 负责后端逻辑、状态流和实现审查。

主要范围：

- AI 教练数据模型定义
- 推导指标处理（体脂率、BMR、BMI、TDEE、目标周期）
- 校准覆盖逻辑
- 首批补采字段结构
- 首日 TODO 的生成输入和状态存储
- 7 天触发与提醒规则
- 第 7 天总结所需的打卡聚合逻辑
- 边界情况、失败状态和集成正确性审查

Claude Code / Codex 是以下内容的 owner：

- 后端契约设计
- 逻辑正确性
- 实现审查
- 系统可靠性

### 10.3 产品协同顺序

三个 Agent 的协同推进顺序应当是：

1. 先由 Antigravity 产出完整的 UI/UX 和前端交互方案。
2. 再由 Claude Code / Codex 将这个前端流映射成数据结构、API 契约和实现逻辑。
3. 最后回到产品视角复核，确认是否满足最初的目标：
   - 教练是主动的
   - 习惯养成是现实可持续的
   - 第 7 天用户能获得明显的成就感

## 11. 立即交付顺序

建议的执行顺序如下：

1. Antigravity：先交付 AI 教练首轮流程的前端方案与实现。
2. Claude Code / Codex：围绕已确定的前端流，定义最小数据模型和后端集成路径。
3. 产品复核：确认这个体验是否真的把“理想身材愿景”转成了“持续行动”。

## 12. 范围说明

本文档定义的是 AI 教练模块的第一期范围：

- 首次主动进入
- 初始身体判断
- 第一批补采
- 首日行动计划
- 首周 7 天节奏

当前不包含：

- 第一周之后的长期陪伴策略
- 更高级的营养个性化
- 付费教练分层
- 教练人格自定义

## 13. PRD 用户故事

### 13.1 首次建联

作为一个刚完成理想身材生成的用户，我希望 AI 教练能先告诉我它已经推断出了什么，而不是立刻问我问题，这样我会更容易产生“它已经理解我”的感觉。

### 13.2 快速启动

作为一个用户，我希望只回答少量高价值问题，而且最好是快速点选，这样我能更轻松地开始，而不是被问卷劝退。

### 13.3 现实节奏

作为一个用户，我希望 AI 教练能给我一个健康、现实的周期建议，并在我选得太激进时做纠偏，这样我不会从一开始就走到不可持续的节奏里。

### 13.4 立即行动

作为一个用户，我希望在完成首次建联后，马上拿到一份今天就能执行的计划，而不是只得到抽象建议。

### 13.5 成就强化

作为一个用户，我希望在第一周结束时，AI 能看见我真正完成了什么，并把它总结出来，让我有继续坚持的动力。

## 14. 验收标准

### 14.1 入口与触发

- 理想身材生成后，首页 AI 机器人能够主动发起首次教练流程
- 首次教练流程以 2-3 条消息完成引导，而不是单条超长内容
- 首页不新增新的主 CTA 入口

### 14.2 初始判断

- 初始身体判断卡能够展示约定的推导指标与一条 AI 阶段判断
- 流程中包含基于“现状与目标差距”的健康理论周期
- 周期建议通过文案表达，不通过按钮文案标注“推荐”

### 14.3 首批补采

- 首批补采必须包含训练基础、伤病/限制、每周训练频率、单次训练时长
- 伤病/限制支持快捷标签和可选文本补充
- 如果用户选择每周 1-2 天，必须触发纠偏并要求重选

### 14.4 首日输出

- 完成首批补采后，同一会话内能立即生成首日计划
- 首日计划必须包含训练卡、饮水/饮食提醒、打卡动作
- 训练任务的粒度足够直接执行，不需要二次解释

### 14.5 首周留存闭环

- 第 1-3 天的消息更主动
- 第 4-7 天根据完成、沉默、连续打卡等信号自适应调整
- 第 7 天必须出现以成就感为核心的周总结，且包含完成次数、用户感受和 AI 认可

## 15. 交付阶段

### Phase A：前端优先探索

负责人：Antigravity

目标：

- 先把“用户能看见和感受到的产品形态”做出来
- 优先验证消息节奏、卡片层级、情绪节奏和信任感

交付物：

- 首轮主动触达流
- 初始身体判断报告卡
- 首批补采交互卡
- 首日 TODO 卡
- 第 7 天成就感总结卡

### Phase B：产品复核与迭代

负责人：产品方向主导，Antigravity 配合迭代

目标：

- 在看到真实前端方案后，再调整交互
- 修正流程顺序、文案语气、卡片信息密度和信任感表达

交付物：

- 修订后的前端交互流
- 固化后的关键交互决策

### Phase C：后端映射与集成

负责人：Claude Code / Codex

目标：

- 将已经确认的前端流映射为最小可用逻辑层
- 避免在交互未稳定前过早重型设计后端

交付物：

- 精简数据模型
- 最小 API 契约
- 触发状态处理
- 推导指标与总结逻辑

## 16. 实施说明

- 这个功能的早期决策，应由前端体验主导。
- 后端工作在前期应保持轻量，等 Antigravity 的方案经过产品复核后，再贴合实现。
- 现阶段做后端规划的目的，是降低后续集成风险，而不是过早锁死产品。

## 17. 页面 / 组件清单

为了方便前端拆活，首期建议明确以下页面内组件：

1. 首页悬浮 AI 机器人主动触达气泡
2. AI 初始身体判断报告卡
3. 健康周期说明消息块
4. 训练基础选择卡
5. 伤病 / 限制双层选择卡
6. 每周训练频率选择卡（含纠偏分支）
7. 单次训练时长选择卡
8. 首日训练任务卡
9. 首日饮水 / 饮食提醒卡
10. 首日打卡动作卡
11. 第 7 天成就感总结卡

这些组件应优先作为一个完整的前端体验链路来设计，而不是零散开发。
