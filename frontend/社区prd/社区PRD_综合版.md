## 1. 背景与定位

RightNow 核心链路已覆盖「看见理想自己（Evolution）」和「AI 教练带练（AI Coach）」。  
社区模块的职责是提供外部驱动力，把“坚持健身”从个人行为升级为“可被看见、可被陪伴、可被拉回”的系统行为。

社区定位：

1. 第一性目标是放大“进步感知”，不是泛社交聊天。
2. 内容中心是“进步报告”，不是碎片化灌水动态。
3. 关系中心是“搭子协作”，不是弱关系加好友。
4. AI 既服务内容生产，也服务关系维护。

## 2. 产品目标与非目标

### 2.1 目标（按优先级）

1. 晒成果：提高用户发布“阶段成果内容”的意愿与质量。
2. 找搭子：提高节奏匹配的搭子建立率与留存率。
3. 激励坚持：通过 AI + 关系协作降低掉队概率。

### 2.2 非目标（当前版本不做）

1. 泛兴趣话题社区（与健身坚持弱相关的内容广场）。
2. 重聊天 IM 系统（已读回执、语音房、复杂群聊等）。
3. 复杂创作者体系（等级、打赏、商业化分成等）。

## 3. 当前实现基线（As-Is）

## 3.1 前端现状

当前 `Community` 页面已上线两标签结构：

1. `广场`：帖子列表、点赞、评论、分页加载、发帖弹窗（文本为主）。
2. `搭子`：按用户 ID 发起邀请、待确认列表、已接受列表、移除关系。

已有交互能力：

1. 发帖（`content`，可扩展 `images/tags`）。
2. 点赞/取消点赞。
3. 展开评论、发送评论。
4. 搭子请求、接受、拒绝/删除。

已存在但尚未接入真实逻辑的视觉模块：

1. AI 社交托管 Banner（Beta 说明文案）。
2. AI 搭子匹配卡（“查看智能推荐”按钮目前未接后端）。

## 3.2 后端现状

已具备可用 API（均需 JWT）：

1. 帖子
   1. `GET /api/posts`（分页）
   2. `GET /api/posts/:id`
   3. `POST /api/posts`
   4. `DELETE /api/posts/:id`（仅作者）
   5. `POST /api/posts/:id/like`
   6. `GET /api/posts/:id/comments`
   7. `POST /api/posts/:id/comments`
   8. `DELETE /api/comments/:id`（仅评论作者）
2. 搭子关系
   1. `GET /api/friendships`
   2. `POST /api/friendships/request`
   3. `PATCH /api/friendships/:id/accept`
   4. `DELETE /api/friendships/:id`

返回规范：

1. 后端统一 `success + data` 包裹。
2. 前端 Axios 拦截器已自动解包为 `data`。

## 3.3 数据模型现状

已落库模型：

1. `Post`：`content/images/tags/likedUserIds`
2. `Comment`
3. `Friendship`：`requesterId/receiverId/status`

当前缺口（与目标 PRD 对比）：

1. 帖子缺少可见范围字段（公开/目标圈层/仅搭子）。
2. 帖子缺少“AI 草稿来源”和“编辑后发布版本”的溯源字段。
3. 缺少“进步报告卡”结构化字段（指标、阶段、对比图类型等）。
4. 缺少“搭子匹配分数/理由”与“双人小房间”模型。

## 4. 目标版本（To-Be）与范围

本 PRD 定义社区模块 `MVP v1`，优先闭环“进步内容 + 搭子关系 + AI 轻运营”。

### 4.1 In Scope（MVP v1）

1. 首页顶部搭子推荐区 + 主区进步动态流。
2. 进步报告卡（标准信息四层）。
3. AI 发帖草稿流（生成草稿 -> 用户编辑 -> 确认发布）。
4. 帖子可见范围（全部 / 相近目标 / 仅搭子）。
5. 搭子推荐与搭子关系管理（先规则匹配，后 AI 增强）。
6. 双人小房间 Lite（状态面板 + AI 共同任务 + 轻互动）。
7. 训练/打卡链路“一键转社区”对接（来自 `TODO和训练记录PRD.md`）。

### 4.2 Out of Scope（MVP v1 不做）

1. 多人房间与群组机制。
2. 完整即时通信系统。
3. 深度内容审核平台（仅基础风控与人工兜底）。

## 5. 核心用户故事

1. 作为用户，我想把阶段成果整理成更有说服力的帖子再发布，而不是手写长文。
2. 作为用户，我希望首页看到“真实进步证据”，从而被激励继续执行计划。
3. 作为用户，我希望系统自动推荐节奏相近的搭子，而非手动盲搜。
4. 作为用户，我希望搭子关系建立后被持续维护，而不是加完好友就结束。
5. 作为用户，我希望训练记录完成后可一键转社区，并在发布前全量可编辑。

## 6. 功能需求（FR）

### 6.1 社区首页与内容流

1. FR-01：首页主视觉必须是“进步动态流”，搭子推荐仅占顶部轻量区域。
2. FR-02：动态流支持分页与“加载更多”，避免重复触发。
3. FR-03：帖子卡默认展示作者、时间、内容摘要、互动数据。
4. FR-04：当帖子具备进步结构数据时，优先渲染“进步报告卡样式”。

### 6.2 进步报告卡

1. FR-05：卡片需支持主视觉（对比图或阶段成果图）。
2. FR-06：卡片需支持 AI 总结标题。
3. FR-07：卡片需支持关键指标（体重变化、连续打卡、训练完成次数等）。
4. FR-08：卡片需支持用户感受文本（可编辑）。

### 6.3 AI 发帖草稿流

1. FR-09：支持用户主动触发“帮我发”。
2. FR-10：支持系统在阶段节点触发“值得发一条”的提醒。
3. FR-11：AI 仅生成草稿，不可直接发布。
4. FR-12：用户发布前必须可编辑文案、图片、可见范围。
5. FR-13：系统需保存“原始 AI 草稿”与“最终发布内容”两份数据用于溯源。

### 6.4 互动能力

1. FR-14：支持点赞/取消点赞。
2. FR-15：支持评论列表与新增评论。
3. FR-16：支持作者删除自己的帖子。
4. FR-17：支持评论作者删除自己的评论。

### 6.5 搭子机制

1. FR-18：首页顶部展示推荐搭子入口。
2. FR-19：搭子匹配优先级为节奏匹配 > 目标匹配 > 风格匹配。
3. FR-20：推荐卡展示目标方向、训练节奏、最近活跃度、推荐理由。
4. FR-21：支持发起邀请、接受邀请、移除关系。

### 6.6 双人小房间 Lite

1. FR-22：每段搭子关系对应一个独立双人房间。
2. FR-23：房间顶部展示双方今日完成状态与连续天数。
3. FR-24：房间中部展示 AI 共同任务（今日任务/鼓励/掉队拉回）。
4. FR-25：房间底部仅提供轻互动（加油、戳一戳、预设短句）。

### 6.7 与训练记录链路集成

1. FR-26：训练记录提交成功后支持“生成反馈卡 -> 转发社区”。
2. FR-27：转发前允许用户全量编辑（文案、图片、结构化指标、可见范围）。
3. FR-28：转发成功后在社区可追溯到来源（训练记录 ID 或反馈卡 ID）。

## 7. 信息架构与页面清单

1. 社区首页（顶部搭子推荐 + 进步动态流）
2. 发帖入口（手动发帖 + AI 帮我发）
3. AI 草稿确认页（编辑后发布）
4. 搭子推荐页（列表与筛选）
5. 双人小房间（状态面板 / AI 引导 / 轻互动）

## 8. 数据与接口设计（MVP 目标态）

## 8.1 数据模型新增建议

1. `Post`
   1. 新增 `visibility`：`public | goal_group | buddies_only`
   2. 新增 `postType`：`normal | progress_report`
   3. 新增 `sourceType`：`manual | ai_draft | training_feedback`
   4. 新增 `sourceRefId`（可选）
   5. 新增 `aiDraftPayload`（JSON，可选）
2. `ProgressSnapshot`（可选独立表，或挂在 Post JSON）
   1. 指标结构（体重变化、打卡连续天数、训练完成次数等）
3. `BuddyRoom`
   1. `friendshipId`（unique）
   2. `statusBoard`（JSON）
   3. `aiGuidance`（JSON）
4. `BuddyInteraction`
   1. `roomId`
   2. `senderId`
   3. `type`：`poke | cheer | template_msg`
   4. `content`

## 8.2 API 演进建议

在保留现有接口兼容的前提下，新增：

1. `POST /api/posts/draft/ai`：生成发帖草稿（不落正式帖子）。
2. `POST /api/posts/from-training/:recordId`：从训练反馈卡生成草稿。
3. `GET /api/friendships/recommendations`：返回候选搭子与匹配理由。
4. `GET /api/buddy-rooms/:friendshipId`：获取双人房间状态。
5. `POST /api/buddy-rooms/:friendshipId/interactions`：发送轻互动。
6. `POST /api/buddy-rooms/:friendshipId/ai-remind`：触发 AI 拉回提醒（系统或定时任务调用）。

## 9. 指标体系（MVP）

1. 内容侧
   1. 进步报告帖占比（`progress_report / all_posts`）
   2. AI 草稿采纳率（草稿发布数 / 草稿生成数）
   3. 社区转发率（训练反馈卡 -> 社区发布）
2. 关系侧
   1. 搭子邀请接受率
   2. 搭子 7 日留存率
   3. 双人房间周活跃率
3. 坚持侧
   1. 建立搭子后 14 日训练完成率变化
   2. 掉队用户被拉回率（提醒后 48h 内恢复打卡）

## 10. 验收标准（UAT）

1. 社区首页进入后，首屏可见搭子推荐和进步动态流。
2. 用户可通过“帮我发”生成草稿，并在确认页编辑后发布。
3. 发布时可选择 3 档可见范围。
4. 训练记录流程可进入“转发社区”并完成发布。
5. 搭子推荐页能展示推荐理由，且可直接发起邀请。
6. 每对已接受搭子关系均可进入独立双人小房间。
7. 小房间可见双方当日状态、AI 共同任务和轻互动入口。

## 11. 里程碑与交付

### Phase 1：内容闭环（1 周）

1. 交付：进步动态流升级、进步报告卡、AI 草稿确认流、可见范围。
2. 验证：完成首批“训练记录 -> AI 草稿 -> 发布”闭环。

### Phase 2：搭子闭环（1 周）

1. 交付：推荐接口、推荐页、邀请/接受/管理优化。
2. 验证：完成“推荐 -> 建立搭子 -> 进入房间”的可用链路。

### Phase 3：关系运营闭环（1 周）

1. 交付：双人小房间 Lite、AI 拉回提醒、轻互动。
2. 验证：监测搭子 7 日留存与掉队拉回效果。

## 12. 风险与依赖

1. 现有“按用户 ID 添加搭子”门槛高，需补充搜索/推荐入口，否则增长受限。
2. `friendships/request` 当前对不存在用户会创建 placeholder 账号，此逻辑仅适合开发态，生产需替换。
3. 社区帖子与训练记录尚未建立强关联，需统一 `sourceRefId` 追溯设计。
4. 进步报告卡依赖 AI 与数据质量，需提供“纯手动降级路径”。

## 13. 角色分工建议（沿用现有协作模式）

1. Antigravity：社区首页、报告卡、草稿确认页、搭子页、房间 UI/交互。
2. Claude Code / Codex：数据模型、API、状态流、边界处理与联调。
3. 用户（产品 owner）：优先级决策、验收标准确认、阶段通过评审。
